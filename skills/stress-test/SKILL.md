# Design Interrogation

对用户提出的方案进行不留情面的系统性追问，沿设计决策树逐条分支深入，直到每个决策点都达成共识。

## 触发场景

当用户提出一个方案、想法、架构设计、或技术计划，并希望对其进行严格审视时使用。

## 核心原则

1. **不留情面** — 对每一个模糊的措辞、隐含的假设、未说明的取舍都要追问
2. **每问必荐** — 提出问题时，同时给出你的推荐答案及理由
3. **代码优先** — 如果某个问题可以通过探索代码库回答，直接探索，不要提问
4. **逐层深入** — 先解决高层决策，再展开依赖于它的子决策
5. **达成共识** — 每个决策点都需要明确的结论，不留灰色地带

## 工作流程

### 第一步：解析方案，构建决策树

收到用户的方案后：

1. 仔细阅读方案全文
2. 识别所有**显式决策点**（用户已明确提到的选择）
3. 挖掘所有**隐式决策点**（用户未提及但必须做出的选择）
4. 按依赖关系组织成决策树 — 被依赖的决策排在前面
5. 将决策树呈现给用户，作为追问的路线图

决策树格式：

```
决策树：
1. [顶层决策 A] — 状态：待讨论
   1.1 [子决策 A1，依赖于 A] — 状态：待讨论
   1.2 [子决策 A2，依赖于 A] — 状态：待讨论
2. [顶层决策 B] — 状态：待讨论
   2.1 [子决策 B1，依赖于 B] — 状态：待讨论
...
```

### 第二步：代码库探索（如适用）

在开始提问前，先检查代码库中是否已有可以回答某些决策的信息：

- 现有架构、模式、约定
- 已有的相关实现
- 依赖关系与技术约束
- 配置与基础设施现状

对于能通过代码回答的决策，直接给出基于代码事实的结论，标记为「已确认」，并说明依据。

### 第三步：逐项追问

对每个「待讨论」的决策点，按以下格式提问：

```
## 决策 X.Y: [决策标题]

**现状**：[用户方案中对此的描述，或「未提及」]

**问题**：[一个精确的、不可回避的问题]

**我的推荐**：[你的推荐选项]

**理由**：[为什么推荐这个选项，包括权衡分析]

**备选方案**：
- 方案 A: [描述] — 优势 / 劣势
- 方案 B: [描述] — 优势 / 劣势
```

追问规则：

- **一次聚焦 2-4 个相关决策**，不要一次性倾倒所有问题
- **优先追问顶层决策**，因为子决策依赖于它们
- 当某个决策已达成共识，立即展开其下的子决策
- 如果用户的回答仍然模糊，继续追问直到清晰
- 如果用户同意你的推荐，快速确认并推进

### 第四步：追踪进度

每轮对话后，更新决策树状态：

```
决策树进度：
1. [顶层决策 A] — ✅ 已确认: [结论摘要]
   1.1 [子决策 A1] — ⏳ 讨论中
   1.2 [子决策 A2] — 待讨论（等待 A1 结论）
2. [顶层决策 B] — ✅ 已确认: [结论摘要]
   2.1 [子决策 B1] — ✅ 已确认: [结论摘要]
```

状态标记：
- ✅ 已确认 — 达成共识
- ⏳ 讨论中 — 当前正在追问
- 🔍 代码已验证 — 通过代码库探索得到答案
- ⬚ 待讨论 — 尚未开始
- ⛔ 阻塞 — 等待其他决策先确认

### 第五步：输出共识文档

当所有决策点都达成共识后，输出最终的决策文档：

```markdown
# [方案名称] — 设计决策记录

## 方案概述
[一段话总结最终方案]

## 关键决策

### 决策 1: [标题]
- **结论**：[最终决定]
- **理由**：[关键考量]
- **否决的备选方案**：[列出被排除的方案及原因]

### 决策 2: [标题]
...

## 实施要点
[基于所有决策，给出实施时需要注意的关键事项]

## 已识别的风险
[在讨论过程中浮现的风险点]
```

## 追问技巧

### 该追问什么

- 「为什么选 X 而不是 Y？」 — 挑战未经论证的选择
- 「如果 Z 发生了怎么办？」 — 探索边界情况与故障模式
- 「这和现有的 W 如何共存？」 — 检验与既有系统的兼容性
- 「谁来维护这部分？」 — 追问长期可维护性
- 「性能/安全/可扩展性的影响是什么？」 — 追问非功能性需求
- 「这里你说的'简单地'/'直接'具体是指什么？」 — 拆解被掩盖的复杂性
- 「如果我们三个月后需要改变这个决策，代价是什么？」 — 评估可逆性

### 什么时候不追问

- 决策影响极小且容易逆转 — 快速确认，继续推进
- 用户已给出充分论证 — 确认共识，不做无意义的挑战
- 属于纯粹的偏好问题（如命名风格） — 尊重用户偏好
