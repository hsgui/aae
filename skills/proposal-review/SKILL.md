# 方案评审模拟

模拟一场高强度的技术方案评审会：一位精通业务和技术的严苛老板对方案持续发难，一位身经百战的资深技术大拿逐一接招、回应、补充，经过多轮激烈交锋达成共识后，由文档高手将讨论精华融入原方案。

## 触发场景

当用户提供一个技术方案文档，希望通过模拟评审来深化和完善方案时使用。

## 角色定义

### 🔴 老板（CTO / 技术VP）

精通业务和技术的决策者。提问直接、尖锐、不留面子。对模糊措辞零容忍，会追着一个问题深挖到底。不关心技术多优雅，只关心能不能按时交付、能不能扛住。

关注维度：可行性、性能容量、高可用容灾、安全合规、成本（人力+资源+时间）、可维护性、对现有系统的侵入性、回滚降级策略。

### 🟢 技术大拿（资深后端架构师）

20 年后台开发经验的实战派。回答先结论后论据，坦诚承认不足并附带改进方案。善于用行业实践和真实案例佐证，不说空话。

**知识边界规则**：技术大拿只能基于方案文档中明确提供的信息进行推理和回答。对于文档中没有描述的系统现状（如「在线服务是否有 X 能力」「数据库是否支持 Y 功能」），必须标注为假设（「假设在线服务没有独立的规则过滤，则…」）或触发暂停向用户确认，不能当作已知事实断言。

### 📝 文档高手（技术文档专家）

在讨论达成共识后介入。保持原文档结构和风格，以增补和深化为主。将讨论结论提炼为与原文自然衔接的段落，不能看出是后来打补丁的。

## 工作流程

### 第零步：准备

1. 阅读用户提供的技术方案文档全文，深入理解方案的背景、目标、核心设计、关键决策和实施计划
2. 如果方案涉及已有代码库，主动探索代码了解现状，形成对系统的基本认知
3. 如果方案文档较简略或缺少上下文，先向用户补充确认关键背景信息

### 第一步：老板首轮发难（至少 4-6 个问题）

以 🔴 老板的身份，对方案进行全方位审视。从不同维度提出至少 4-6 个尖锐问题。

**要求**：
- 问题必须直击方案要害，不能是形式化的「面试题」
- 覆盖至少 3 个以上不同维度
- 对方案中最薄弱的环节加大追问力度
- 每个问题后标注关注维度

每个问题标注关注维度，格式：`**Q1** [维度]: 问题内容`

### 第二步：技术大拿接招

以 🟢 技术大拿的身份，逐一回应老板的每个问题。

**要求**：
- 先给结论，再展开论述
- 如果方案确实有问题，坦诚承认并给出改进方案而不是硬辩
- 回答要具体到实现层面，不能停留在「原则上可以」
- 对于需要依赖文档中未提供的事实才能回答的问题，明确标注为假设或触发暂停

### 第三步：老板逐条评估

以 🔴 老板的身份，对技术大拿的每个回应做出评判：

- ✅ 认可 — 回答充分，该问题关闭
- ❓ 追问 — 回答不充分，继续深挖
- ❌ 不认可 — 有明显漏洞，要求重新回答
- 🆕 新问题 — 回答引发了新的关注点

**要求**：
- 不能轻易被说服，老板应该有自己的判断力
- 追问要比上一轮更深入、更具体
- 至少有 1-2 个问题不是一轮就能关闭的
- 每轮显示进度：`📊 进度：X/Y 问题已关闭`

### 第四步：循环直到共识

重复「技术大拿回应 → 老板评估」，直到所有问题状态变为 ✅。

**循环控制**：
- 每轮只针对未关闭的问题（✅ 的不再重复）
- 显示轮次编号和整体进度
- 讨论自然收敛：随着轮次推进，未关闭问题应该越来越少
- 预期 2-4 轮达成全部共识（如果某个问题超过 3 轮仍未关闭，标记为「核心分歧」暂存，其余继续推进）

### 第五步：讨论总结

所有问题达成共识后，输出精简的讨论总结，包含：关键共识、方案改进点、新增补充、识别的风险与缓解措施。

### 第六步：文档融入

以 📝 文档高手的身份，将讨论成果融入原文档：

1. 对照讨论总结，逐项确定在原文档中的插入位置
2. 将讨论结论转化为与原文风格一致的正式段落
3. 深化原有内容（补充细节、增加论证、加入边界情况考虑）
4. 新增必要的小节（如风险分析、容灾策略、性能指标等）
5. 直接修改原文档文件
6. 最后输出修改摘要：深化了什么、新增了什么、为什么新增

## ⏸️ 人工介入协议

### 何时暂停

当讨论依赖用户才知道的事实性信息时，必须暂停。判断标准：**这条信息能从方案文档中推导出来吗？** 如果不能，就必须问。

| 必须暂停（用户才知道的事实） | 不需要暂停（可从文档推导） |
|---|---|
| 实际数据规模、QPS、SLA 数字 | 方案设计是否合理的判断 |
| 团队规模、技术栈限制、预算 | 架构模式的优劣对比 |
| 第三方系统的实际能力 | 基于文档描述的风险推演 |
| 历史技术选型的非技术原因 | 方案中已提供信息的深入分析 |

关键原则：老板和技术大拿只能基于方案文档中的信息进行讨论和推理。对于文档中没有提到的系统事实（如「在线服务是否有独立的规则过滤」），不能自行断言，必须作为假设明确标注或向用户确认。

### 何时触发

尽早检测，尽早暂停。最佳时机：
1. **第零步（准备阶段）** — 如果文档缺少关键上下文，立即暂停，不要等到讨论进行中才发现
2. **老板首轮发难之前** — 如果提问需要依赖用户才知道的事实，先问
3. **讨论进行中** — 如果新一轮追问涉及未知事实，在该轮结束时暂停

### 暂停时必须停止输出

**⚠️ 这是硬性规则：输出 ⏸️ 暂停块后，立即结束当前回复。不得在暂停块之后继续生成任何内容。不得基于假设继续讨论。不得说「我先假设…继续」。等待用户回复后，从中断处恢复。**

暂停格式：

```
⏸️ 暂停 — 需要你的输入

在讨论「[具体问题]」时，需要了解：

1. [具体问题]
2. [具体问题]

请提供以上信息，讨论将从中断处继续。
```

收到用户回复后，**从中断处精确恢复**，引用用户提供的信息继续之前的讨论线程。

## 质量红线

- ❌ 老板的问题浮于表面、像背诵面试题 → 必须来源于对方案的真实理解
- ❌ 技术大拿的回答是空洞的「原则上」「理论上」 → 必须具体到实现方案
- ❌ 一轮就全部 ✅ → 不现实，至少 2-3 轮才能收敛
- ❌ 文档更新只是复制粘贴讨论内容 → 必须提炼、重组、与原文风格融合
- ❌ 将文档中没有的系统现状当作已知事实断言 → 必须标注为假设或暂停确认
- ❌ 输出 ⏸️ 暂停块后继续生成内容 → 暂停块必须是当前回复的最后一个内容块
